# AI Project Intelligence Platform

> Turn a raw project idea into a complete, connected enterprise execution plan — PRD, feasibility analysis, business intelligence, ROI modeling, org planning, and roadmap — generated by a coordinated team of AI agents and kept consistent as your assumptions change.

**Status:** Pre-build / Phase 0 (core AI reasoning loop under validation) — see [Current Phase](#current-phase) below.

---

## What This Is

This is not a "PRD generator." It's an AI-powered Project Intelligence Platform that behaves like a fused team of a **Product Manager + Business Analyst + Solution Architect + Strategy Consultant**, working over a persistent, project-scoped conversation.

A user brings a raw idea — a sentence, a chat conversation, images, PDFs, supporting documents — and the platform converts it into a full, interconnected set of enterprise planning artifacts:

1. **PRD** — problem statement, goals, personas, user stories, requirements, acceptance criteria, success metrics, MVP definition
2. **Feasibility Analysis** — technical, business, and operational feasibility, risks, limitations
3. **Business Intelligence** — market analysis, competitor analysis, SWOT, business model suggestions
4. **ROI / Financial Planning** — development, infrastructure, and team cost estimates, revenue potential, break-even ranges
5. **Enterprise Resource Planning** — required departments, roles, skills, team structure, hiring roadmap
6. **Execution Roadmap** — milestones, timeline, priority sequencing, delivery strategy

The defining architectural property — and the whole point of the product — is that **these six outputs stay connected**. Change one assumption (e.g., "assume we only have $50K, not $500K") and the ROI model, hiring plan, and roadmap all re-derive accordingly, instead of becoming six independent, contradictory documents.

---

## Why This Exists

Today, going from "I have an idea" to "I have a fundable, buildable, staffable plan" requires a PM, a business analyst, a financial modeler, an HR/ops person, and usually a consultant — each producing a separate document, often inconsistent with the others, over several weeks. Most early-stage teams skip most of this and pay for it later: they build the wrong thing, understaff it, or run out of runway because costs were never modeled realistically.

This platform compresses that entire chain into a guided AI workflow, while keeping the rigor (and the connectedness across documents) that a real planning team would maintain.

## Core Architectural Idea

```
User (idea, chat, files)
        ↓
Next.js — UI, auth, session management
        ↓
FastAPI — business logic, authorization, orchestration trigger
        ↓
LangGraph — stateful multi-agent workflow engine
        ↓
AI Agents — Input Understanding, Clarification, Document/Image,
            PRD, Market Research, Competitor Analysis,
            Technical Architecture, ROI/Finance, HR Planning,
            Risk Analysis, Roadmap, Final Report
        ↓
MongoDB (system of record) + Vector Store (RAG, later phases)
        ↓
Connected, versioned enterprise reports
```

Full detail on every layer, every agent, and every collection lives in [`plan.md`](./plan.md).

**The single most important design decision in the whole system:** every generated report is a derived view of one shared, evolving project context object — not an independently generated document. This is what makes cross-document consistency possible, and it's why the AI orchestration layer (LangGraph, not a single-shot prompt) exists at all.

## Tech Stack

- **Frontend:** Next.js, TypeScript, Tailwind CSS
- **Authentication:** Auth.js / NextAuth.js (Google + GitHub OAuth), MongoDB Adapter

- **Database:** MongoDB

- **Infrastructure:** Docker, CI/CD, cloud deployment

Rationale, trade-offs, and where simpler chains are preferred over full LangGraph orchestration are covered in [`plan.md`, Section 15](./plan.md#15-important-engineering-guidance).

## Current Phase

**Phase 0 — Foundation & Validation.**

Before any authentication, database, or frontend work begins, the core reasoning loop (raw idea → clarification → structured PRD) is being validated in isolation using the prompts and test idea set in [`phase0_agent_prompts.md`](./phase0_agent_prompts.md). Nothing in Phase 1 begins until this loop reliably scores well against real, diverse project ideas — see the exit criteria and rubric in that document.

### Roadmap at a glance

| Phase | Focus | Exit Signal |
|---|---|---|
| **Phase 0** | Validate the AI core loop | Clarification questions are sharp, not generic; PRDs are genuinely useful across diverse test ideas |
| **Phase 1 — MVP** | End-to-end single-user loop: auth, projects, chat, file upload, reduced agent set, basic reports | A first-time user can go from idea to useful PRD without confusion |
| **Phase 2 — Production** | Full 12-agent suite, organizations/teams, incremental re-derivation, report editing, background jobs | Assumption changes correctly propagate only to dependent reports; team collaboration works with enforced permissions |
| **Phase 3 — Enterprise** | SSO, advanced governance, billing maturity, integrations, scale optimization | Passes security/compliance review for target enterprise customers; multi-tenant isolation explicitly verified |

Full detail, build sequences, team sizing, and risks per phase are in [`phase.md`](./phase.md).

## Guiding Principles for Contributors

1. **The clarification step is the product.** A generic clarification prompt produces generic everything downstream — treat prompt quality here as a first-class engineering concern, not a copywriting afterthought.
2. **Never treat LangGraph state as durable storage.** MongoDB is the permanent system of record; workflow state is scoped to a single run.
3. **Every authorization check happens server-side, against a cryptographically verified identity.** Never trust client-supplied `user_id` or resource IDs.
4. **Structured outputs everywhere.** Every agent returns schema-validated JSON, not free text — this is what makes cross-report consistency and incremental re-derivation possible.
5. **Don't build ahead of validated need.** RAG, custom permission overrides, SSO, and bespoke integrations are explicitly deferred until real usage or real customer demand justifies them — see the "explicitly out of scope" section of each phase in `phase.md`.
